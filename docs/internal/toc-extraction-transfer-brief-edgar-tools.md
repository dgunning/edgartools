# TOC / Section Extraction — Transferable Knowledge for edgar.tools

**Audience:** the edgar.tools codebase, which implements its own SEC filing
section-extraction (it does **not** depend on edgartools as a library).
**Purpose:** transfer what edgartools learned the hard way about table-of-contents
extraction — the algorithms, the failure modes, and the heuristics — so edgar.tools'
independent implementation inherits the lessons without re-paying the discovery cost.
**Nothing here is an API call.** Everything is a concept to reimplement.
**Origin of these lessons:** edgartools' `edgar/documents/utils/toc_analyzer.py` and the
design sprint `docs/internal/parser-design-sprint-2026-05.md`.

---

## 1. The problem shape

SEC filings (10-K, 10-Q, 20-F, ...) carry a Table of Contents whose entries link to
anchors elsewhere in the document. "Extracting Item 1A" means: find the anchor the TOC
entry for Item 1A points to, find the anchor for the *next* section, and return the
content between them. This sounds trivial and is not — the difficulty is entirely in
(a) reliably mapping TOC entries to the right anchors and (b) cleanly slicing HTML
between two anchors. Both fail *silently* (wrong content, no error) on a predictable
set of filings. Silent wrong-content is the worst failure class for a data product;
design every decision below to make failures loud or at least introspectable.

---

## 2. Lesson 1 — Filing agents are the single best structural signal

SEC filings are generated by a handful of **filing agents** (vendors), each of which
leaves a distinctive signature and produces a structurally consistent TOC. Detecting
the agent first, then parsing with agent-specific logic, is far more reliable than one
universal parser. Empirical distribution (edgartools' survey of 200 10-Ks, early 2026):

| Agent | Share | Signature (in first ~3KB of HTML) | TOC characteristics |
|---|---|---|---|
| Workiva | ~35% | string `Workiva` | UUID anchors (`#i7193…_13`); `<a>` split across 3 cells sharing one href; item # only in *combined cell text*, never the anchor |
| Donnelley / DFIN | ~26% | `DFIN`, `Donnelley`, `dfinsolutions` | semantic anchors (`#item_1_business`); one `<a>` per cell; heading often `INDEX`; text-only `PART I` rows set part context |
| Toppan Merrill | ~11% | `Merrill`, `Toppan` | descriptive anchors with numeric suffix (`#ITEM1BUSINESS_392371`); split cells; **zero-width spaces (U+200B)** and nbsp in text |
| Novaworks | ~9% | `ThunderDome`, `Field: Set; Name: xdx;` | short anchors (`#item1a`, `#Item1C`); inconsistent casing; **Item 1 often shares `#part1`**; page numbers in `<a class="tocPGNUM">` |
| (others) | <5% each | `CompSci`, `Certent`, `Broadridge`, `EDGARsuite`, `SEC Publisher` | — |
| unknown | ~14% | — | falls to a generic link-scan |

**Transferable takeaways:**
- Detect the agent from the **first ~3000 characters** (signatures live in HTML
  comments, meta tags, copyright notices) and check in frequency order for early exit.
- Build the four high-volume agent parsers first — they cover ~81% reliably.
- The **generic fallback is where silent failures concentrate.** Treat anything that
  came through it as lower-trust and worth a sanity check (§6).
- Item-number recovery has three independent sources, used in priority order:
  **(1) preceding table cell** (`<td>Item 1.</td><td><a>Business</a></td>`),
  **(2) the anchor ID pattern**, **(3) the link text**. Different agents make different
  ones reliable (Workiva → text only; DFIN → anchor only), so try all three.

## 3. Lesson 2 — Do not bake 10-K shape into "universal" code (the core debt)

This is edgartools' single biggest architectural mistake, and the most valuable thing
to *not* repeat. 10-K-shaped assumptions leaked into supposedly form-agnostic code:
- a hard cap of 15 on "bare" item numbers (page-number cells above 15 were rejected —
  but a 10-Q has only Items 1–6, so a `<td>8</td>` page cell became a phantom "Item 8");
- a text-keyword table ("Financial Statements" → Item 8, "Business" → Item 1) that is
  correct *only* for 10-K — on a 10-Q a "Notes to Condensed... Financial Statements"
  link became a bogus Item 8; on a 20-F the whole table is wrong;
- a sort-order table with the same 10-K assumptions.

Every form since (10-Q, 20-F, 40-F, S-1, DEF 14A) was then bolted on as `if form ==
"10-Q": ...` branches. The branch count only grows.

**Transferable design rule:** make form-awareness **data, not branches.** Keep a
declarative per-form schema — the canonical item vocabulary, the part structure (10-Q
has Part I *and* Part II, each with its own Item 1!), and the bare-item ceiling — and
let one form-agnostic parser consult it. The parser is mechanism; the schema is the only
place form knowledge lives. If edgar.tools adds any section-shaping logic, model it this
way from day one.

**Corollary — part structure matters for 10-Q.** A 10-Q has `Part I Item 1` *and*
`Part II Item 1` (different content). Keys must be part-qualified (`part_i_item_1`,
`part_ii_item_1`). Track "current part" statefully while scanning the TOC: part headers
often appear as standalone text-only rows (`PART II`) with no link, sometimes many rows
before the items they govern.

## 4. Lesson 3 — Slicing HTML between two anchors: the algorithm and its seven traps

Given the correct start and end anchors, returning clean HTML between them is its own
hard problem. edgartools' working algorithm:

1. Walk the document in order. Turn collection **on** at the start-anchor element, **off**
   at the end-anchor element. Collect every element seen while on.
2. Reduce to **top-level** collected elements — those whose parent is *not* also
   collected. Serializing an element already emits its full subtree, so serializing a
   nested descendant *in addition to* its ancestor emits that descendant (and any table
   under it) multiple times. (edgartools shipped a bug where one section returned a table
   **24×**; the fix is this top-level-only rule.)
3. **Repair orphaned table fragments.** When anchors sit inside a `<table>`, the
   top-level collected elements can be bare `<tr>`/`<td>`/`<tbody>`. Most HTML parsers
   **silently drop a bare `<tr>`** (invalid outside a table) — losing the rows *and any
   tables inside them*. Re-wrap each run of table-internal fragments in a minimal
   `<table><tbody>…` before re-parsing.
4. Wrap the result in a single container element so downstream code has one well-formed
   root to parse or render.

The **seven edge cases** that break naïve implementations (keep this as a test checklist):
1. next-section heading leaks in via nested anchors,
2. start/end share a wrapper (need lowest-common-ancestor reasoning),
3. start anchor == end anchor (Novaworks Item 1 shares `#part1`) — guard it,
4. inline anchor wrappers (anchor is mid-paragraph, not a block boundary),
5. last-section-in-wrapper leaks trailing content,
6. table-row-bounded anchors lose `<table>`/`<tbody>` wrappers (→ step 3),
7. nested-table re-serialization duplicates tables (→ step 2).

Cases 6 and 7 have clean fixes (above). Cases 1/2/4/5 stem from the end anchor being
reached *after* some adjacent content was already collected (the anchor is nested deeper
than the section's container). The principled fix is **renderer-aware boundaries** — the
extractor hands the renderer a precise element range and the renderer never emits past
it — rather than slicing text and regex-stripping the bleed-in afterward (see §5).

## 5. Lesson 4 — Fix boundaries upstream, don't regex-strip them downstream

edgartools tried to clean "page number + PART header + next Item" bleed-in at section
boundaries with a regex. That regex grew through **11 rounds of review** to handle plain,
markdown-escaped, heading-decorated, bold-decorated, and combined variants — each round
caught a new shape. The lesson: a cleanup regex *chases* renderer output forever. The
renderer already knows where the section ends; make it stop there. Spend the effort on
precise boundaries (§4), not on stripping artifacts after the fact.

---

## 6. Lesson 5 — The cheap guardrail that catches silent wrong-content

The highest-leverage safety net edgartools wishes it had from the start: an **expected
content-size band per (form, item)**. It catches both catastrophic directions:
- **Too small** (< ~200 chars for a content item like 1, 1A, 7, 7A, 8, 9A, 15): the
  anchor probably landed on a `PART` header instead of the item body — extract the real
  item by searching for the literal `ITEM N. TITLE` heading nearby instead.
- **Too large** (megabytes): boundary overshoot — the slice ran past the next section.
  edgartools had a 10-K return **1.78MB of raw HTML** this way.

When content falls outside the band, **lower the confidence and attach a warning the
caller can read** — do not return it silently as high-confidence. Pair this with a
confidence + detection-method label on every extracted section so consumers (especially
RAG pipelines) can distinguish a high-trust TOC extraction from a low-trust regex guess.

## 7. The known-bad regression corpus

These specific filings exercise the catastrophic failure modes. Any independent
section-extraction implementation should regression-test against them:

| Filing | Failure mode it triggers |
|---|---|
| **Goldman Sachs 10-K** | Part I / Part II item misclassification; "Business" silently returned 668KB of MD&A |
| **Citigroup 10-K** | boundary overshoot → 1.78MB of raw HTML markup returned |
| **JPMorgan 10-K** | large-bank structure that defeats anchor mapping (edgartools only handles it via a legacy fallback) |
| **PPG 10-Q** (`0000079879-26-000170`) | page-number cell (`<td>8</td>`) became phantom "Item 8" — the 10-K-cap-on-a-10-Q bug |

The GS and Citi cases are **mapping/boundary bugs, not slicing bugs** — a perfect HTML
slicer still returns wrong content if handed the wrong anchors. They live or die on
Lessons 2 and 6, not Lesson 4.

## 8. Quick reference — the transferable rules

| Lesson | Rule |
|---|---|
| Agent detection | Detect filing agent from first ~3KB; build the 4 high-volume parsers first (§2) |
| Item recovery | Try preceding-cell → anchor-ID → link-text, in that order (§2) |
| Form shape | Form-awareness is a per-form schema (data), never `if form ==` branches (§3) |
| 10-Q parts | Part-qualify keys; track current part statefully across text-only rows (§3) |
| Anchor slicing | Top-level-only serialization + orphaned-row repair; the 7-case checklist (§4) |
| Boundaries | Precise boundaries upstream beat regex-stripping downstream (§5) |
| Safety net | Expected size-band per (form, item) → flag outliers, don't return silently (§6) |
| Confidence | Label every section with detection method + confidence (§6) |
| Regression | Test against GS / Citi / JPM / PPG (§7) |
