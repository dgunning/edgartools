# Industry-Specific Extraction Strategy and Rules

This document outlines how industry-specific rules are defined, structured, and expanded within the `edgartools` codebase.

## 1. Defining Industries and Rules

The codebase uses a multi-layered approach to handle industry specifics, separating configuration from logic.

### A. Industry Definitions (`industry_mappings.json`)

Industries are centrally defined in `edgar/entity/data/industry_mappings.json`. This file maps Standard Industrial Classification (SIC) codes to internal industry keys.

**Example Structure:**

```json

"banking": {

"name": "Banking & Financial Services",

"sic_ranges": [[6020, 6029], [6211, 6211]],

"extension_file": "banking.json",

"description": "Commercial banks, savings institutions"

}

```

### B. Metric Mappings (`industry_metrics.yaml`)

High-level mappings between standard metrics (e.g., `COGS`, `OperatingIncome`) and industry-specific counterparts are defined in `edgar/xbrl/standardization/config/industry_metrics.yaml`.

**Example (Banking):**

```yaml

banking:

concept_mapping:

COGS:

counterpart: InterestExpense

xbrl_concepts: ["InterestExpense", "InterestExpenseTotal"]

OperatingIncome:

counterpart: PPNR

calculation: "NetInterestIncome + NonInterestIncome - NonInterestExpense"

```

### C. Detailed Concept Extensions (`industry_extensions/*.json`)

Granular, data-driven concept mappings are stored in `edgar/entity/data/industry_extensions/`. These files (e.g., `banking.json`) contain occurrence rates and structural relationships for XBRL concepts specific to that industry. These are often generated by analysis scripts.

### D. Complex Extraction Logic (`industry_logic/__init__.py`)

For rules that cannot be expressed as simple mappings (e.g., conditional logic, "if A exists use A else B+C"), Python classes are used.

* **Location:** `edgar/xbrl/standardization/industry_logic/__init__.py`

* **Mechanism:** Subclasses of `IndustryExtractor` (e.g., `BankingExtractor`, `SaaSExtractor`) implement methods like `extract_short_term_debt` or `extract_operating_income`.

* **Features:**

* **Dual-Track Extraction:** Calculates metrics using multiple methods (e.g., "Street View" vs "Economic View") for validation.

* **Guardrails:** Includes sanity checks (e.g., "Cash < 1% of Assets" implies restricted cash).

## 2. Strategy for Expanding Knowledge

The strategy for expanding coverage to new industries or refining existing ones is documented in `sandbox/notes/`, specifically `004_ai_agent_concept_mapping` and `008_bank_sector_expansion`.

### Workflow

1. **Gap Detection**: Identify industries or companies where standard extraction fails or yields insufficient data.

2. **Definition**: Add the industry to `industry_mappings.json` with appropriate SIC ranges.

3. **Data Analysis**:

* Run extraction tests on a cohort of companies (e.g., "Commercial Mega Banks", "Regional Banks").

* Use tools like `learn_mappings.py` to generate initial extension files.

4. **Logic Implementation**:

* If simple mapping is insufficient, create a new `IndustryExtractor` subclass in `industry_logic`.

* Implement specific methods for difficult metrics (typically Debt, Revenue, Operating Income).

5. **Validation**:

* **Multi-Path Verification**: Compare "Direct Tag" vs "Calculated" values.

* **Reference Checks**: Verify against external data sources (e.g., yfinance) as seen in usage of `extract_short_term_debt_yfinance`.

* **Accounting Identity**: Ensure `Revenue - Expenses ≈ Operating Income`.

### Future Direction (AI Agent)

Notes in `004_ai_agent_concept_mapping` propose an AI-driven agent to handle the long tail of unmapped concepts, moving away from simple string similarity towards semantic analysis and automated validation against expected metrics.
